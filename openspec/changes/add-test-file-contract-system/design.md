## Context

PhysicsGuard is used in two different ways:

- model-only or blueprint work, where there may be no external test data file;
- test-bench/test-file work, where a concrete file or export contains many fields, time-series data, commands, measurements, setpoints, flags, calibration values, and metadata.

The new route is for the second case only. It is a sibling route to existing AI debugging, signal mapping review, preflight, and closure skills. It must not make ordinary PhysicsGuard audits heavier when there is no test data file.

## Goals / Non-Goals

**Goals:**
- Require one machine-checkable resolved contract per test data file.
- Generate file facts with scripts, not AI narration: file hash, format, field count, names, row/sample counts, time range, frequency, continuity, and extractor identity.
- Check both directions of completeness: model needs data, and file fields need explicit disposition.
- Track schema drift across files, test-bench versions, extraction scripts, aliases, field signatures, and model bindings.
- Gate broad AI analysis claims on file contract pass/fail/partial evidence.
- Add a direct Codex skill route that is installed locally and validated against repository source.

**Non-Goals:**
- No universal requirement that every PhysicsGuard project use test-file contracts.
- No full database adapter or commercial-tool adapter in the first implementation.
- No automatic physical-model generation from test data.
- No hidden conversion engine; conversion evidence stays explicit and reviewable.
- No high-fidelity physical models or commercial solver equivalence claims.

## Core Concepts

### DataFileManifest

`DataFileManifest` is generated by an extractor script. It records objective file facts:

- source file path, content hash, size, format kind, delimiter/encoding/header data where applicable;
- field count, row count, sample count, field names, units, data types, missing counts, and optional min/max summaries;
- time column, start/end time, duration, nominal sample rate, sampling mode, continuity, and gaps;
- extractor script path, script hash, config hash, generated time, and extractor version.

The manifest is evidence. Users should not hand-edit generated manifest output except in narrowly documented fixture tests.

### TestFileContract

`TestFileContract` is the per-file resolved contract. It binds:

- file identity and manifest hash;
- test-bench profile;
- extractor profile;
- model binding;
- parameter catalog;
- parameter role matrix;
- parameter mapping edges;
- coverage policy;
- optional dataset segments;
- known file defects and safe-claim limits.

The contract may reuse shared profiles, aliases, and model bindings, but the resolved check must be file-specific.

### Parameter Coverage

Parameter coverage separates identity from roles:

- `ParameterCatalog` records stable source field identity.
- `RoleMatrix` records test-bench role, physical role, model role, owner block, verification role, and coverage status.
- `MappingEdges` record relationships such as `same_quantity_as`, `command_for`, `readback_of`, `boundary_for`, `parameter_for`, `derived_from`, `aggregate_of`, `checked_by`, `gates_mode_for`, and `excluded_with_reason`.
- `CoveragePolicy` decides which findings are fail, partial, warning, or info.

### ModelBinding

`ModelBinding` connects the contract to PhysicsGuard audit artifacts:

- hierarchy/audit path and hash;
- observed template or observed mapping source;
- PhysicsGuard package version;
- compatible test-bench profiles or manifest signatures;
- stale rules when model, hierarchy, version, or expected fields change.

## File-Level Contract Flow

```text
test data file
  -> extractor script
  -> DataFileManifest
  -> TestFileContract
  -> parameter coverage check
  -> model binding check
  -> contract pass/partial/fail
  -> hierarchy evaluate / closure only if claim boundary allows it
```

## CLI Shape

Add a `testfile` command group:

```powershell
python -m physicsguard.cli testfile manifest DATA_FILE --profile PROFILE --out MANIFEST
python -m physicsguard.cli testfile inspect CONTRACT --pretty
python -m physicsguard.cli testfile contract-check CONTRACT --pretty
python -m physicsguard.cli testfile project-check INDEX --pretty
python -m physicsguard.cli testfile diff OLD_CONTRACT NEW_CONTRACT --pretty
```

Add a lower-level `coverage` group:

```powershell
python -m physicsguard.cli coverage check CONTRACT --pretty
```

## FlowGuard Plan

Add `.flowguard/physicsguard_test_file_contract_model.py` with blocks:

- `GenerateDataFileManifest`
- `VerifyManifestFreshness`
- `ResolveTestbenchProfile`
- `ResolveModelBinding`
- `BuildParameterCatalog`
- `ApplyRoleMatrix`
- `ResolveMappingEdges`
- `EnforceCoverageContract`
- `GateAnalysisClaim`

Key invariants:

- no contract pass without manifest;
- no manifest without extractor identity;
- no pass when source file, field signature, extractor, or model binding is stale;
- no source field missing from catalog;
- no catalog entry without a role;
- no role without disposition;
- no mapping to unknown model target;
- no broad claim without passing test-file contract when test data is in scope.

## Codex Skill Route

Add `physicsguard-test-file-contract-review` as an installed Codex child skill. It is conditionally required only when test data files are present. Update the main AI debugging skill to dispatch to this route for test-bench data, but preserve ordinary model-only routes.

## Risk Controls

- Keep core extractor dependency-light. Use CSV/TSV and existing YAML/JSON manifest support by default.
- Treat manifest counts and time/frequency facts as script evidence, not AI-derived claims.
- Treat coverage pass as coverage evidence only; physical residual correctness still requires PhysicsGuard reports and closure.
- Preserve peer-agent changes and rerun validation after later writes.
